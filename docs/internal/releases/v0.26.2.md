# v0.26.2 Release Notes

Release name: The Living Story Preview

Release date: 2026-04-27

Primary surfaces:

- Storefront home page
- Storefront Our Why page
- Storefront collection page
- Admin dashboard
- Admin release announcement dialog
- Admin changelog page
- Internal changelog documentation

## Executive Summary

v0.26.2 is a customer-facing polish and release-communication pass. The release
replaces the static Our Why page with a more interactive preview, makes the home
page hero easier to read, removes decorative card icons that were fighting with
the saree imagery, and adds an admin changelog so the team can inspect release
history from the admin shell.

This release intentionally does not redesign the collection filter system. The
filter UI was lightly simplified earlier, but the full planning pass is deferred
so it can be treated as a dedicated catalog-navigation project rather than a
side effect of the Our Why work.

## Product Intent

The product goal is to move From the Trunk away from static, text-heavy brand
pages and toward story-led commerce. Grace and Diya want Our Why to feel more
alive, with the future direction including voiceover, slides, image-led pacing,
and a more novel narrative format.

This release creates the first preview of that direction:

- A chaptered Our Why experience that behaves like a guided story.
- Browser voiceover controls for the active chapter.
- Slide-style navigation that can be extended into richer media.
- Stronger visual hierarchy so the page carries emotion without becoming a wall
  of text.

## Storefront Changes

### Our Why

The `/why` route now renders a new interactive experience instead of the previous
static content. The new component lives in
`components/sections/our-why-experience.tsx`.

The experience includes:

- Chapter navigation for the story flow.
- Image-led chapter cards.
- Voiceover playback through the browser speech API.
- A calmer, less manifesto-heavy content structure.
- Progressive enhancement so the page still reads coherently without voice
  playback.

The current implementation is a preview layer. It establishes the interaction
model and editorial rhythm before final voice recording, custom audio assets, or
more advanced slideshow mechanics are added.

### Home Hero

The home hero was updated in `components/sections/hero-section.tsx` to improve
visibility on the garden image.

The main changes are:

- Stronger overlay treatment on the image.
- Better immediate readability for the headline and support copy.
- Removal of the above-the-fold reveal dependency so critical hero content is
  visible as soon as the page renders.
- Cleaner call-to-action contrast.

This directly addresses the visual QA note that the first viewport looked
beautiful but the headline and supporting copy were too hard to read.

### Featured Collection Cards

The decorative overlay icons were removed from
`components/sections/featured-collection.tsx`.

Reasoning:

- The icons competed with product photography.
- The icons made the cards feel more generic and less premium.
- The saree imagery, title, fabric, price, and CTA are the important scan
  targets.

The cards now rely on image, type, spacing, and product content instead of
symbolic decoration.

### How It Works Cards

The decorative step icons were removed from
`components/sections/how-it-works.tsx`.

The cards retain the step numbers, imagery, title, and explanation. This keeps
the flow legible while removing the visually loud icon layer from the customer
surface.

### Bento Grid Icon Handling

`components/ui/bento-grid.tsx` now treats card icons as optional. This avoids a
future regression where a card system requires an icon even when the design
direction calls for a quieter image-led layout.

## Admin Changes

### Release Data Model

`lib/admin/releases.ts` now stores richer release data.

Each release can include:

- Version number.
- Release name.
- Release date.
- Announcement visibility.
- Demo link and demo label.
- Summary.
- Highlight cards.
- Detailed grouped changes.

The grouped changes power the dashboard latest-update card, the release
announcement dialog, and the changelog page. Keeping this data in one module
prevents the release notes from drifting across admin surfaces.

### Clickable Version Badge

`components/admin/app-version-badge.tsx` now renders as a link by default and
opens `/admin/changelog`.

The same badge is reused across:

- Admin sidebar.
- Admin top bar.
- Admin mobile menu.
- Dashboard latest-update card.
- Release announcement dialog.

The component still supports `href={null}` for read-only contexts, such as
inside the changelog page where the badge is informational instead of
navigational.

### Dashboard Latest Update

`app/(admin)/admin/page.tsx` now includes a richer latest-update card.

The dashboard card shows:

- Current release name.
- Clickable version badge.
- Short summary.
- First item from each release change group.
- Link to the full changelog.

This gives admins a quick update surface without forcing them to open the full
release dialog.

### Release Announcement Dialog

`components/admin/release-announcement-dialog.tsx` now includes detailed release
notes, not just highlight cards.

The dialog now communicates:

- What changed.
- Why it matters.
- Which grouped fixes are included.
- Where to preview the release.

Seen state remains per version through local storage, so v0.26.2 can show once
for each admin without hiding future releases.

### Admin Changelog

`app/(admin)/admin/changelog/page.tsx` is a new admin route for release history.

The page includes:

- Current release summary.
- Highlight cards.
- Full grouped release notes.
- Previous release history.
- Release demo links.

This is the durable in-app changelog surface requested for admins. The internal
Markdown changelog remains the engineering evidence record.

## Versioning

- `package.json`

## Copy And Punctuation Cleanup

Customer-facing em dashes, en dashes, and visible dot separators were removed
from checked storefront surfaces.

Areas checked include:

- Account order detail and wishlist copy.
- Cart and checkout copy.
- Collection page and product detail copy.
- How It Works, Our Story, policy, and terms pages.
- Search bar copy.
- Product card copy.
- SEO JSON-LD strings.
- Customer email templates.

This keeps the storefront voice calmer and avoids the punctuation style Abe
called out as unwanted on customer-facing surfaces.

## Deferred Work

### Collection Filter Redesign

The collection filter needs a dedicated design pass. The current page is
functional, but the ideal next pass should plan the information architecture
before changing controls.

Recommended next questions:

- Should filters be edit-first, fabric-first, occasion-first, price-first, or a
  mixed system?
- Which filters are shopper-facing, and which are internal curation logic?
- Should the filter panel collapse into a drawer on mobile?
- Should sort controls be separated from curation filters?
- Should selected filters become removable chips above the product grid?
- How many products should be visible before pagination or progressive loading
  is introduced?

The decision to defer this prevents a half-redesign from landing in the same
release as the Our Why and changelog work.

### Final Our Why Media

The new Our Why experience is a preview. The future media pass can add:

- Recorded voiceover from the brand team.
- More precise story beats from Grace and Diya.
- Timed slide progression.
- Audio captions or transcript controls.
- Richer visual transitions once the final content direction is approved.
